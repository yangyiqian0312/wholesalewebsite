# AGENTS.md

本项目技术栈：

- 前端：Next.js
- 后端：Fastify
- 数据库：PostgreSQL

## 项目结构

- `app/`：Next.js 页面与前端应用结构
- `components/`：可复用前端组件
- `docs/`：产品、交互、设计与项目说明文档
- `prototypes/`：原型、静态预览和草稿页面
- `backend/`：Fastify 后端服务
- `backend/src/`：后端源码
- `backend/src/routes/`：后端路由
- `backend/src/modules/`：按业务拆分的模块
- `backend/src/integrations/`：第三方集成，例如 Inflow
- `backend/src/lib/`：通用工具与基础能力
- `backend/src/db/`：数据库访问与数据层相关代码
- `backend/prisma/`：PostgreSQL schema 与迁移文件

如果后续目录结构需要调整，应优先在现有结构上做清晰演进，不要频繁推翻重来。

## 核心原则

- 先满足需求，再写代码。
- 代码保持简洁、直接、易读。
- 尽量避免重复代码。
- 可以复用的逻辑或组件，优先抽取并复用。
- 不要用“打补丁”式方式修问题，要优先选择结构清晰、可维护的方案。
- 不要擅自增加未被要求的功能、配置、交互、字段或抽象。
- 写代码时要一步一步做，先完成当前明确范围，再进入下一步。
- 不要一次性把联想到的所有事情都做完，然后留下大量以后再补的坑。
- 每一步实现都应尽量是可运行、可验证、可继续迭代的。

## 实现要求

- 写代码前先理解现有实现和上下文，避免凭空改造。
- 优先沿用项目中已有的模式、目录结构和组件设计。
- 如果一个能力已经可以通过已有组件、工具函数或通用模块实现，优先复用，不重复造轮子。
- 抽象必须有明确收益；不要为了“看起来更通用”而过度封装。
- 修改时应从根因入手，避免只在表面追加特殊判断。
- 保持单个组件和函数职责清晰，避免臃肿。
- 实现时优先做最小完整闭环，不要为了未来可能会做的需求预埋大量半成品。
- 如果一个较大目标需要拆分，应先完成第一步的完整落地，再继续后续步骤。
- 不要留下明显未完成但已接入主流程的占位逻辑、假接口或半成品结构，除非我明确同意。
- 做 UI 时优先保持结构扁平、直接，不要出现明显的“一个框套一个框”式写法；如果必须嵌套，必须有明确的布局或交互理由。

## Inflow API 约束

- Inflow 是上游系统，不是前台页面的实时数据库。
- 官方文档当前说明的 rate limit 是每分钟 `60 requests per minute`。
- 任何读写 Inflow 的实现都必须优先考虑这个限流，不允许默认把每个用户访问都直接转成一次 Inflow 请求。
- 读取 Inflow 时应优先采用分页、缓存、请求去重、重试退避和本地落库等方式，减少直接请求次数。
- 面向用户的前台页面应优先读取本地数据库或本地缓存，而不是直接依赖 Inflow 实时响应。
- 如果后续要做写入 Inflow 的逻辑，必须先停下来问我，再继续实现。
